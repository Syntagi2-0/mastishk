# Persistence layer

## Purpose

The Syntagi Lite persistence layer is a modular-monolith mapping over the Flyway-owned PostgreSQL
schema. Entities contain persistence state and small domain invariants; repositories contain only
data-access operations. Controllers and complete application workflows are intentionally outside
this layer.

## Module ownership

| Module | Entities and repositories |
|---|---|
| `auth` | `User`, `UserRepository` |
| `business` | `Business`, `BusinessRepository` |
| `staff` | `BusinessUser`, `BusinessUserRepository` |
| `customer` | `Customer`, `CustomerRepository` |
| `servicecatalog` | `BusinessService`, `ServiceSchedule` and their repositories |
| `appointment` | `AppointmentSlot`, `Appointment` and their repositories |
| `queue` | `QueueSession`, `QueueToken` and their repositories |
| `notification` | `Notification`, `NotificationRepository` |
| `dashboard` | No source entities; future purpose-specific projections/read models |
| `common` | `BaseEntity` and persistence-neutral exception infrastructure |

## Aggregate boundaries

- `User` and `Business` are independent roots. `BusinessUser` joins them without cascading either
  lifecycle.
- `Customer` is a business-scoped root and never references `User`, because MVP customers do not
  authenticate.
- `BusinessService` is the service-catalogue root. `ServiceSchedule` is its configuration child,
  but persistence code still avoids `CascadeType.ALL`.
- `AppointmentSlot` protects capacity. `Appointment` references a slot optionally and remains an
  operational record if a slot is later removed.
- `QueueSession` owns queue counters and current-token coordination. `QueueToken` is independently
  persisted operational history and owns the optional appointment reference.
- `Notification` references its context but never owns or cascades customer, appointment, token, or
  business lifecycles.

## Relationship design

Relationships are unidirectional and lazy by default. The model does not expose large collections
such as `Business.services`, `QueueSession.tokens`, or `Appointment.queueToken`. This keeps aggregate
loading explicit, prevents accidental cascade operations, and avoids serializing deep object graphs.

Known list workflows use entity graphs where their immediate related state is required. Other
workflows can add projections or purpose-specific fetch joins. Relationships must not be changed to
eager loading as a workaround for transaction-bound access.

The nullable `QueueSession.currentToken` mapping is a lazy `ManyToOne`, matching the actual
non-unique `current_token_id` foreign key. Sessions are inserted with no current token, tokens are
inserted afterward, and the session is updated last. This avoids a circular insert dependency.

## Base entity and auditing

`BaseEntity` maps UUID identity, `OffsetDateTime` audit timestamps, and a nullable `Long` optimistic
version before first persistence. UUIDs are generated by Hibernate with `GenerationType.UUID`; the
database `gen_random_uuid()` default remains compatible with non-Hibernate inserts. Spring Data JPA
auditing populates `created_at` and `updated_at`, while PostgreSQL defaults protect direct inserts.

Entity equality uses non-null immutable identity and Hibernate's effective class. No entity uses
Lombok `@Data`, public ID setters, or audit-field setters.

## Locking and concurrency

### Queue sessions

`QueueSessionRepository.findByIdForUpdate` obtains `PESSIMISTIC_WRITE`. Future `NEXT` and counter
allocation workflows must use this lock in one transaction. Counter increment methods require an
open session and return the new value. `@Version` remains a secondary lost-update guard for ordinary
updates.

The next-token candidate query filters to `WAITING`, then orders by:

1. priority descending;
2. non-null scheduled time before null scheduled time;
3. scheduled time ascending;
4. queue order ascending.

A candidate and a pessimistically locked candidate variant are available. The application service
will apply remaining eligibility rules, use a one-row page, and perform the current-token transition
within the locked session transaction.

### Appointment-slot capacity

`AppointmentSlotRepository.findByIdAndBusinessServiceIdForUpdate` obtains `PESSIMISTIC_WRITE` before
capacity mutation. `reserveOne()` rejects blocked/closed or full slots. `releaseOne()` rejects a
negative result. The database `booked_count` checks provide the final integrity boundary.

### Uniqueness races

The application must treat database uniqueness as authoritative for case-insensitive users and
business slugs, business-scoped customers/services, service/date queue sessions, appointment
tokens, token display labels, and queue order. A pre-check improves feedback but never replaces
handling the insert-time constraint violation.

## Queue-token state machine

Allowed transitions are:

```text
WAITING -> CALLED | CANCELLED | NO_SHOW
CALLED  -> COMPLETED | SKIPPED | CANCELLED
SKIPPED -> CALLED | CANCELLED | NO_SHOW | WAITING
```

Terminal states reject further transitions. `InvalidQueueTokenTransitionException` maps to the
standard conflict response. `tokenDisplay` is constructor-assigned and has no public mutation
method; queue ordering remains a separate field.

## Repository query responsibilities

- Case-insensitive lookup methods follow the functional indexes for user email and business slug.
- Business-scoped repositories query through relationship IDs without loading the parent first.
- Confirmed appointment queries order by scheduled start time and can exclude appointments that
  already have queue tokens using `NOT EXISTS`.
- Schedule queue-opening queries return active weekday candidates with service and business loaded;
  timezone and midnight calculations remain in Java.
- Queue repositories expose live-token reads, waiting/skipped counts and lists, deterministic next
  candidates, and pessimistic-lock variants.
- Notification reads support pending delivery order and lookup by appointment or queue token.

Repositories do not implement workflow decisions or mutate unrelated aggregates.

## Flyway ownership and Hibernate validation

Flyway migrations are the only schema-definition mechanism. `spring.jpa.hibernate.ddl-auto` is
`validate`, so application startup fails on table, column, type, or nullability mismatches without
creating or changing schema objects. Functional and partial indexes remain solely in Flyway because
JPA annotations cannot represent them accurately.

PostgreSQL Testcontainers integration tests run Flyway before Hibernate validation and exercise
repository queries and database constraints. They are skipped only when the executing environment
has no Docker-compatible runtime; unit-level entity state tests still run without infrastructure.

## Future extraction

The module ownership table is also the intended microservice extraction boundary. Unidirectional
references and repository ownership make dependencies visible. A future extraction will replace
cross-module foreign keys with stable UUID references plus APIs/events; no shared generic repository
layer needs to be dismantled first.
